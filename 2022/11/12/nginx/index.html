http {
    include       /etc/nginx/mime.types; #把别的配置文件引入到这个主配置文件里，mime.type是一个解析文件，告诉浏览器什么样的后缀名需要用什么格式解析
    default_type  application/octet-stream; #如果说mime.type中不包含的类型，则使用application/octet-stream这种流的形式传输给浏览器

    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';

    access_log  /var/log/nginx/access.log  main;

    sendfile        on; #数据0拷贝
    #tcp_nopush     on;

    keepalive_timeout  65;#保持连接，超时时间

    #gzip  on;

    include /etc/nginx/conf.d/*.conf; #把/etc/nginx/conf.d目录下所有conf结尾的配置文件加载进来
    
}
  • sendfile：数据0拷贝
    • 如果把这个关了，列如一个用户来下载一个视频，nginx需要先把视频读取到内存中，然后复制给网络接口，通过网络接口，发送给客户端。
    • 如果开了这个，nginx不执行读取内存这个步骤，直接给网络接口发送一个信号sendfile(文件描述符)，网络接口读取这个文件直接发送给客户端

2.虚拟主机与域名解析

现在很多服务器的性能非常好，以前一个服务器只能对应一个域名，非常的浪费资源。现在有了nginx之后，我们可以使用多个域名同时解析到同一个服务器的ip。我们只需要在nginx中配置多个虚拟主机，设置规则判断是通过哪个域名来访问我们服务器，让这些请求进入到对应的资源目录下即可。

2.1域名

  • @表示解析域名本身
  • *表示任意二级域名，列如，www.lvfeiy.top，test.lvfeiy.top等

2.2虚拟主机

  • 一个server_name 可以填写多个域名，可以使用通配符,列如
server {
    listen       80; #监听的端口号
    server_name  *.lfymall.com www.lfy.com ;#当前这台虚拟主机的主机名，通常是域名
}

2.2.1）location匹配规则

模式 含义
location = /uri 精确匹配，只有完全匹配上才能生效
location ^~ /uri 对url路径进行前缀匹配，如果不带^匹配优先级会在正则匹配之后，带^则会优先于正则匹配
location ~ pattern 表示区分大小写的正则匹配
location ~* pattern 表示不区分大小写的正则匹配
location /uri 不带任何修饰符，也表示前缀匹配，但是匹配优先级在正则之后
location / 通用通配符，任何未匹配到其他location规则的请求都会匹配到 /，相当于默认

2.2.2）location匹配优先级

  1. 精确匹配 =
  2. 前缀匹配 ^~
  3. 按文件中顺序的正则匹配
  4. 匹配不带任何修饰符的前缀匹配
  5. 交给 / 通用匹配

匹配示例

#精确匹配
location = / { 
   echo "规则A";
}
location = /login {
   echo "规则B";
}
#优先于正则的前缀匹配
location ^~ /static/ {
   echo "规则C";
}
location ^~ /static/files {
    echo "规则X";
}
#正则匹配（区分大小写）
location ~ \.(gif|jpg|png|js|css)$ {
   echo "规则D";
}
#正则匹配（不区分大小写）
location ~* \.png$ {
   echo "规则E";
}
#前缀匹配（不加^~优先级低于正则匹配）
location /img {
    echo "规则Y";
}
#通用匹配（以上规则匹配不到时进入此默认规则）
location / {
   echo "规则F";
}

2.2.3）定义规则

location / {
   #当路径匹配进来后，root就代表了让这个请求到/usr/share/nginx/html这个路径去寻找资源  而proxy_pass 代表让请求转到某某域名下
       root   /usr/share/nginx/html;
       index  index.html index.htm; #默认页
   }
  • root :表示去哪里寻找资源
  • index表示默认页名字，如果请求 /进入此规则，意思就是去/usr/share/nginx/html目录下找index.html或index.htm
http {
    #upstream 定义一组负载均衡地址
    upstream lfymall{ 
        server 192.168.1.4; 
    }
    include /etc/nginx/conf.d/*.conf;
 server {
     isten        80
     server_name    lfymall.com
 location /test {
        proxy_pass http://baidu.com
  }
  location /test2 {
        proxy_pass http://lfymall
  }
  location /test3{
        proxy_pass http://192.168.158.139
  	}
}
}

  • upstream 表示定义一组负载均衡地址，名字自定义

  • proxy_pass 表示请求进入之后跳转到某个地址，这个地址可以是一个网址，也可以是ip http://192.222.22.xx 也可以是一个使用upstream自定义一组负载均衡的地址

    • 需要注意的是，如果跟上的地址是顶层域名，也就是例如http://lvfeiyt.top 不包含前缀的域名，会重定向。如果是二级域名则不会。发生重定向后，后面的请求将不走nginx

2.3负载均衡

上述所说upstream 定义一组需要负载均衡的地址

upstream lfymall{ 
    server 192.168.1.4 ; 
    server 192.168.2.4 ; 
}
upstream lfymall1{ 
    server 192.168.1.4 weight=8; 
    server 192.168.2.4 weight=2; 
    server 192.168.3.4 weight=1 backup; 
}
  • 默认什么都不加为轮循
  • weight：权重，也就是负载均衡访问到目标ip的。比例，访问192.168.1.4的比例次数概率为8/8+2
  • backup：备用机，正常情况下不负载均衡到此ip。当192.168.2.4宕机时启动备用服务器
    • 实际上backup用的比较少，因为如果说服务器出现宕机，备用机的内容和主应用服务器应该是相同的，那么就算备用机顶上去也会出现各种原因引发的宕机

以下属于只需了解的负载均衡规则

  • ip_hash：根据客户端的ip地址转发同一台服务器，可以保持会话。目前分布式session状态问题已经被springSession配合redis解决，而且主流趋势也趋向于使用jwt token等方式，所以用的比较少
  • least_conn：最少连接访问
  • url_hash：根据用户访问的url定向转发请求
  • fair：根据后端服务器的响应时间转发请求，需要安装插件，容易流量倾斜，所以也用的少

2.4动静分离

首先需要让请求经过nginx进行反向代理，也就是 客户端 - -> nginx 转发–> tomcat服务器。当请求经过nginx时，写一个匹配规则，如果说是一个静态资源的请求，让这个请求直接去nginx的某个目录寻找资源

server {
    listen       80; #监听的端口号
    server_name  www.lvfeiy.top;#当前这台虚拟主机的主机名，通常是域名
	location /static{
		root /usr/share/nginx/html; #让静态请求进入/usr/share/nginx/html寻找资源
	}


    location / {
        root   /usr/share/nginx/html;
        index  index.html index.htm; #默认页
    }

  • 比如一个tomcat服务器的地址为192.111.222.33 ，nginx服务器的地址为110.110.11.11。那么我只需要解析一个域名www.lvfeiy.top指向nginx的ip。

    nginx在设定规则，是转发给tomcat服务器。还是直接让请求进入nginx目录寻找资源。

    如果发起的请求是一个动态api请求，我写一个匹配规则proxy_pass http://192.111.222.33 让这个请求转发给tomcat服务器。

    如果发起的请求是一个静态请求，我写一个匹配规则 root /usr/share/nginx/html 让请求直接从nginx服务器中寻找资源，而不去tomcat服务器中寻找。

  • 这个请求会自己根据路径寻找资源。比如

    <script src="/static/js/xxx.js"></sctipt>
//那么/代表根路径，会自动带上当前的域名或ip。正常情况下这个请求地址应该是http://192.111.222.33/static/js/xxx.js。而我们是通过nginx反向代理指向的请求
//也就是说我们通过htpp://www.lvfeiy.top反向代理到了192.111.222.33服务器。所以地址为http:/www.lvfeiy.top/static/js/xxx.js。根据这个路径写一个前缀匹配规则location /static。请求会进入/usr/share/nginx/html寻找 /static/js/xxx.js  。也就是这样不再去tomcat中寻找资源，直接从nginx中返回。完成了动静分离

  • nginx的反向代理只会转发ip。也就是只会改变请求对应ip地址。然后拼接上原路径

2.5UrlRewrite

用于隐藏后端api接口

rewrite语法格式及参数语法

    location /spitPage {
      rewrite ^/([0-9]+).html$ /spitPage?pageNum=$1 break; #$1代表匹配第一个正则规则的值
      proxy_pass http://192.111.11.11
    }

# 原本的后端接口为地址为spitPage 参数为pageNum 。通过rewrite写一个正则表达式规则，可以让地址变为http://192.111.11.11/数字.html。相当于取了一个别名。让nginx帮我们转发。隐藏了后端真实api 。


flag标记说明：

last #本条规则匹配完成后，继续向下匹配新的location URI规则

break #本条规则匹配完成即终止，不再匹配后面的任何规则

redirect #返回302临时重定向，浏览器地址会显示跳转后的URL地址

permanent #返回301永久重定向，浏览器地址栏会显示跳转后的URL地址

2.6防盗链

    location /test {
        valid_referers www.lvfeiy.top; #设置可以访问静态资源的Referer
        if ($invalid_referer) {
        return 403;
        }       
        root   /usr/share/nginx/html;
        
    }
#valid_referers none www.lvfeiy.top
none， 如果Referer 头域不存在也可以访问。但是如果有referer就必须正确，否则无法访问。
blocked，检测 Referer 头域的值被防火墙或者代理服务器删除或伪装的情况。这种情况该头域的值不以
“http://” 或 “https://” 开头。
server_names ，设置一个或多个 URL ，检测 Referer 头域的值是否是这些 URL 中的某一个。

  • 当使用动静分离时。配置valid_referers www.lvfeiy.top 。可以防止用户直接通过浏览器 http://www.lvfeiyt.top/xxx/xxx.jpg这种形式访问静态资源

    而通过反向代理的前端展示图片不受影响

  • 原理：比如打开www.lvfeiy.top这个网址。然后页面初始化的时候加载静态资源。发送给nginx的静态资源请求中会带有一个Referer: http://www.lvfeiy.top 表示当前打开网页的域名。而如果是直接通过http://www.lvfeiy.top/static/xxx.jpg这种形式的访问是不会带有Referer请求头的。

    或是别人的网页放置了我们的资源。比如别人网页是http://www.test.com/。页面初始化发送请求给我们的nginx服务器。但是这时候别人的Referer是www.test.com。和我们设置的可通过的referer规则不一致就会返回403。这样设置的规则就避免了别人盗用静态资源

2.6.1）防盗链给用户展示自定义错误页面

展示403错误码略微有点潦草。如需要给用户一个友好的提示。

   location /test {
    valid_referers www.lvfeiy.top; #设置可以访问静态资源的Referer
    if ($invalid_referer) {
    return 403;
    }       
    root   /usr/share/nginx/html;
    
}
 error_page   403  /error/403.html;
	 location = /error {
  	  root   /usr/share/nginx/html;
}
  • 只需要让 403错误码 错误页指向 /error/403.html 然后被一个前缀匹配的location/error规则捕获。让其到/usr/share/nginx/html目录下寻找/error/403.html。即可完成自定义页面展示

2.61）防盗链给用户展示自定义错误图片

如果不想展示一个html想展示一个图片可以这么来做

   location /test {
    valid_referers www.lvfeiy.top; #设置可以访问静态资源的Referer
    if ($invalid_referer) {
    rewrite ^/    /img/fdl.jpg break; # ^/正则表示匹配所有地址。  让其请求地  址变为/img/fdl.jpg  break浏览器不显示跳转地址
    } 
       root   /usr/share/nginx/html; #然后去/usr/share/nginx/html寻找/img/fdl.jpg
    
}

3.linux应用服务器防火墙

  • 开启防火墙

    systemctl start firewalld

  • 重启防火墙

    systemctl restart firewalld

  • 重载规则

    firewall-cmd --reload

  • 查看已配置规则

    firewall-cmd --list-all

  • 指定ip访问和端口

    firewall-cmd --permanent --add-rich-rule="rule family="ipv4" source address="192.168.44.101"
port protocol="tcp" port="8080" accept"
    • 添加一个可信任地址192.168.44.101 ，并开放tcp连接端口8080

  • 移除规则

    firewall-cmd --permanent --remove-rich-rule="rule family="ipv4" source
address="192.168.44.101" port port="8080" protocol="tcp" accept"

4.nginx高可用keeplivated

下载

yum install openssl-devel #安装依赖    
yum install keepalived #安装keeplicated

如果是docker安装的nginx 。也一样。docker的nginx暴露出哪个端口就通过keepalived暴露的ip加端口。也就是说keepalived是通过帮我们把多台网卡的ip更换成一个固定ip

每一台nginx都安装keeplivated，它会帮我们检测对方是否存活。如果宕机了会使用ip漂移到另一台机器，也就是说我们只需要连接keeplivated的ip，然后把组分好即可

配置在/etc/keepalived/keepalived.conf

global_defs {
router_id lb111 #自定义id
}

vrrp_instance lfy {
state MASTER #名字可以自定义
interface ens33 #网卡名
virtual_router_id 51 
priority 100 #权重值，存活多台nginx的时候谁来当主机
advert_int 1 #检测存活的时间
#在内网中组成一组需要让auth_type 和auth_pass的值相同       
authentication {
auth_type PASS
auth_pass 1111
}
#虚拟ip。只需要外界连接这个ip即可
virtual_ipaddress {
192.168.139.111
}
systemctl start keepalived #启动

启动两台虚拟机启动docker的nginx容器和keepalived。外界ping keepalived的ip。然后关机一台权重较高的虚拟机。出现短暂延时，依然可以ping通
