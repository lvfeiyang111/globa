Reids 入门
Redis

1.Redis单线程还是多线程

问：Redis是单线程还是多线程

答：Redis4之后才慢慢支持多线程，直到Redis6/7后才稳定。

​ 版本3.x单线程

​ 版本4.x，负责处理客户端请求的线程是单线程，但是开始加了点多线程的东西，比异步删除（持久化）

​ 版本6.0.x及2022年7.0版本后，使用多线程

Redis是单线程主要是指网络IO和键值对读写是由一个线程来完成的，Redis处理客户端请求时包括获取（socket读）、解析、执行、内容返回（socket写）等

都由一个顺序串行的主线程处理，这就是所谓的“单线程”。

但是Redis其他功能，比如持久化RDB、AOF、异步删除、集群数据同步、都是由额外的线程执行的

Redis命令工作线程是单线程，但是，整个Redis来说，是多线程的。

问：Redis为什么快

  1. 基于内存操作：Redis所有数据都是存在内存中，因此所有运算都是内存级别的，所以性能搞
  2. 数据结构简单：Redis的数据结构是专门设计的，而这些简单的数据结构的查找和操作的时间大部分都是O(1)，因此性能高
  3. 多路复用和非阻塞I/O：Redis使用I/O多路复用来监听多个socket连接客户端，这样可以使用一个线程来处理多个连接
  4. 避免上下文切换：因为是单线程模型，因此避免了不必要的上下文切换和多线程竞争，这就省去了多线程切换带来的时间性能上的消耗

问：Redis为什么放弃单线程

  • 比如：正常情况下del可以很快删除数据，但是当被删除的key是一个非常大的对象时，那么del就会造成Redis主线程卡顿，由于Redis是单线程的，需要等待很久这个线程才会释放，所以4.0引入了惰性删除

2.Redis-多线程IO复用

对于Redis主要的性能瓶颈是内存或网络带宽而并非cpu

Redis6/7中，使用了多个IO线程来处理网络请求，提高网络请求的处理并行度

但是，Redis的多IO线程只是用来处理网络请求的，对于读写命令仍然用的单线程，这是因为Reids处理请求时，网络处理经常是瓶颈，通过多个IO线程并行处理网络操作，可以提升实例的整体处理性能，而继续使用单线程执行命令操作，就不用为了保证Lua脚本、事务的原子性，而外开发多线程互斥机制。

主线程：建立连接请求Socket———–>将socket放入全局等待队列—————>以轮循的方式将Socket连接分配给IO线程————>主线程阻塞

IO线程：将Socket和线程绑定———–>读取Socket中的请求并解析——————->请求解析完成——————>让主线程执行请求命令

2.1五种IO模型

BIO：阻塞IO

NIO：非阻塞IO

IO multiplexing：IO多路复用

signal driven IO：信号驱动IO

asynchronous IO：异步IO

IO multiplexing

IO：操作系统层面数据在内核态和用户态之间的读写操作

多路：多个客户端连接

复用：复用一个或多个线程

IO多路复用：一个或一组线程处理多个TCP连接，使用单进程就能同时处理多个客户端连接

实现IO多路复用的模型有三种：select、poll、epoll三个阶段来描述

如果在实际应用中，发现redis实例CPU开销不大但吞吐量没有提升，可以考虑使用Redis7多线程机制，加速网络处理

#redis.conf
#如果你有4核cpu建议开启2or3个,如果8核cpu建议6个
io-threads 3
#启动多线程IO
io-threads-do-reads no

3.BigKey

问：为什么生产环境禁止使用key *

答：因为Redis命令读写是单线程的，数据量大的情况keys *将会导致Redis阻塞，key * 是一个遍历操作，时间复杂度为O（n）

*如何禁用key 、flushdb、flushall等命令

#1043行
rename-command keys ""
rename-command flushdb ""
rename-command flushall ""

scan

#scan cursor [MATCH pattern] [Count count]
# cursor 游标 pattern 匹配的模式 count 返回的数量

#扫描后回返回游标，游标为0代表遍历完一遍，默认count为10
scan 0 match k* count 15
  • 基于游标的迭代器，需要基于上一次游标延续之前的迭代过程，以0作为游标开始一次新的迭代，知道命令返回游标0完成一次遍历，不保证每次执行都返回某个给定数量的元素，支持模糊查询，一次返回的数量不可控，只能是大概率符合count参数。

多大的key,value算BigKey？

String类型控制在10kb以内，hash、list、set、zset元素个数不要超过5000

  • string是value，最大512MB，但是超过10kb就算bigkey
  • hash、list、set、zset元素个数超过5000就是bigkey

BigKey有哪些危害？

  • 内存不均，集群迁移困难
  • 超时删除、大key删除阻塞主线程
  • 网络流量阻塞

BigKey是如何产生的？

  • 社交类：比如某博主的粉丝，典型逐步递增
  • 汇总统计：某个报表，年积月累

BigKey如何发现？

#-i --每隔多久休眠一次，防止scan阻塞，但是扫描时间可能变长
redis -cli -a 密码 -p 端口 --bigkeys -i 0.1

Memory Usage：计算某个key的字节数

#计算某个key的字节数
MEMORY USAGE key

BigKey如何删除？

非字符串的bigkey，不要使用del删除，使用hscan、sscan、zscan方式渐进式删除，同时要注意防止bigkey过期时间自动删除问题（自动过期慢查询中不会出现，但latency可查）。

hdel key k1 k2 k3

总之，scan查，分批删内容，最后删key

redis.conf配置文件LAZY FREEING

  • 阻塞和非阻塞删除
lazyfree-lazy-eviction no
lazyfree-lazy-expire no
lazyfree-lazy-server-del yes
replica-lazy-flush yes
lazyfree-lazy-user-del yes
  • lazyfree-lazy-eviction

针对redis内存使用达到maxmeory，并设置有淘汰策略时；在被动淘汰键时，是否采用lazy free机制；因为此场景开启lazy free, 可能使用淘汰键的内存释放不及时，导致redis内存超用，超过maxmemory的限制。此场景使用时，请结合业务测试。（生产环境不建议设置yes）

  • lazyfree-lazy-expire

针对设置有TTL的键，达到过期后，被redis清理删除时是否采用lazy free机制；此场景建议开启，因TTL本身是自适应调整的速度。

  • lazyfree-lazy-server-del

针对有些指令在处理已存在的键时，会带有一个隐式的DEL键的操作。如rename命令，当目标键已存在,redis会先删除目标键，如果这些目标键是一个big key,那就会引入阻塞删除的性能问题。此参数设置就是解决这类问题，建议可开启。

  • slave-lazy-flush

针对slave进行全量数据同步，slave在加载master的RDB文件前，会运行flushall来清理自己的数据场景， 参数设置决定是否采用异常flush机制。如果内存变动不大，建议可开启。可减少全量同步耗时，从而减少主库因输出缓冲区爆涨引起的内存使用增长。

  • lazyfree-lazy-user-del yes
# 翻译概要：将代码中的del替换成unlink可能不是那么太容易，所以为了让del
# 命令的实际效果等价于unlink，你可以修改配置文件为lazyfree-lazy-user-del yes

lazy free的监控

lazy free能监控的数据指标，只有一个值：lazyfree_pending_objects，表示redis执行lazy free操作，在等待被实际回收内容的键个数。并不能体现单个大键的元素个数或等待lazy free回收的内存大小。所以此值有一定参考值，可监测redis lazy free的效率或堆积键数量；比如在flushall async场景下会有少量的堆积。

4.1Canal

监听mysql变动，Cannal模拟mysql从节点，去解析binlog日志

https://github.com/alibaba/canal

传统Mysql主从复制原理：

  1. 当master主服务器上数据发生改变，则将其改变写入二进制事件日志文件中
  2. slave从服务器会在一定时间间隔内对master主服务器上的二进制日志进行探测，探测其是否发生改变，如果探测到master主服务器的二进制事件日志发生改变，则开始一个IO Thread请求master二进制事件日志；
  3. 同时master主服务器为每个IO Thread启动一个dump Thread ，用于向其发送二进制事件日志；
  4. slave收到二进制事件日志保存到自己本地中继日志文件中；
  5. slave将启动SQL Thread从中继日志中读取二进制日志，在本地重放，使其与主服务器数据一致

canal工作原理：

  1. canal模拟MySQL slave交互协议，伪装成slave，向Mysql发送dump协议
  2. Mysql收到dump请求，开始推送 binlog日志给slave（canal）
  3. canal解析binlog

package com.lfy;
import java.net.InetSocketAddress;
import java.util.List;


import com.alibaba.otter.canal.client.CanalConnectors;
import com.alibaba.otter.canal.client.CanalConnector;
import com.alibaba.otter.canal.common.utils.AddressUtils;
import com.alibaba.otter.canal.protocol.Message;
import com.alibaba.otter.canal.protocol.CanalEntry.Column;
import com.alibaba.otter.canal.protocol.CanalEntry.Entry;
import com.alibaba.otter.canal.protocol.CanalEntry.EntryType;
import com.alibaba.otter.canal.protocol.CanalEntry.EventType;
import com.alibaba.otter.canal.protocol.CanalEntry.RowChange;
import com.alibaba.otter.canal.protocol.CanalEntry.RowData;
public class CanalClient {
    private final static String CANAL_IP = "127.0.0.1";
    public static void main(String args[]) {
        // 创建链接
        CanalConnector connector = CanalConnectors.newSingleConnector(new InetSocketAddress(CANAL_IP,
                11111), "example", "", "");
        int batchSize = 1000;
        int emptyCount = 0;
        try {
            connector.connect();
            //默认所有库的所有表
//            connector.subscribe(".*\\..*");
            /**
             * 
             * 订阅库全表
             * connector.subscribe("test\\..*");
             * 单表
             * connector.subscribe("test.user");
             * 多规则组合
             * connector.subscribe("test\\..*,test2.user1,test3.user2");
             */
            connector.subscribe("canal.canal");
            connector.rollback();
            //监听多久
            int totalEmptyCount = 120;
            while (emptyCount < totalEmptyCount) {
                Message message = connector.getWithoutAck(batchSize); // 获取指定数量的数据
                long batchId = message.getId();
                int size = message.getEntries().size();
                if (batchId == -1 || size == 0) {
                    emptyCount++;
                    System.out.println("empty count : " + emptyCount);
                    try {
                        Thread.sleep(1000);
                    } catch (InterruptedException e) {
                    }
                } else {
                    emptyCount = 0;
                    // System.out.printf("message[batchId=%s,size=%s] \n", batchId, size);
                    printEntry(message.getEntries());
                }

                connector.ack(batchId); // 提交确认
                // connector.rollback(batchId); // 处理失败, 回滚数据
            }

            System.out.println("empty too many times, exit");
        } finally {
            connector.disconnect();
        }
    }

    private static void printEntry(List<Entry> entrys) {
        for (Entry entry : entrys) {
            if (entry.getEntryType() == EntryType.TRANSACTIONBEGIN || entry.getEntryType() == EntryType.TRANSACTIONEND) {
                continue;
            }

            RowChange rowChage = null;
            try {
                rowChage = RowChange.parseFrom(entry.getStoreValue());
            } catch (Exception e) {
                throw new RuntimeException("ERROR ## parser of eromanga-event has an error , data:" + entry.toString(),
                        e);
            }

            EventType eventType = rowChage.getEventType();
            System.out.println(String.format("================&gt; binlog[%s:%s] , name[%s,%s] , eventType : %s",
                    entry.getHeader().getLogfileName(), entry.getHeader().getLogfileOffset(),
                    entry.getHeader().getSchemaName(), entry.getHeader().getTableName(),
                    eventType));
            System.out.println(entry);

            for (RowData rowData : rowChage.getRowDatasList()) {
                if (eventType == EventType.DELETE) {
                    printColumn(rowData.getBeforeColumnsList());
                } else if (eventType == EventType.INSERT) {
                    printColumn(rowData.getAfterColumnsList());
                } else {
                    System.out.println("-------&gt; before");
                    printColumn(rowData.getBeforeColumnsList());
                    System.out.println("-------&gt; after");
                    printColumn(rowData.getAfterColumnsList());
                }
            }
        }
    }

    private static void printColumn(List<Column> columns) {
        for (Column column : columns) {
            System.out.println(column.getName() + " : " + column.getValue() + "    update=" + column.getUpdated());
        }
    }
}

                    System.out.println(rowData);
                    redisInsert(rowData.getAfterColumnsList(), tableName);
                } else {
                   /* System.out.println("-------&gt; before");
                    redisUpdate(rowData.getBeforeColumnsList(),tableName);
                    System.out.println("-------&gt; after");
                    redisUpdate(rowData.getAfterColumnsList(),tableName);*/
                    redisUpdate(rowData, tableName);
                }
            }
        }
    }

 
    private void redisInsert(List<CanalEntry.Column> columns, String key) {
        if (!CollectionUtils.isEmpty(columns)) {
            // 插入操作

            String id = null;
            JSONObject jsonObject = new JSONObject();
            for (CanalEntry.Column column : columns) {
                if (column.getIsKey()) {
                    id = column.getValue();
                }
                jsonObject.put(column.getName(), column.getValue());
            }
            ;
            RedisUtil.set(getKey(key).concat(id), jsonObject);
            log.info("插入==============================================,k:{},v:{}", getKey(key).concat(id), jsonObject);
        }
    }

    private void redisUpdate(CanalEntry.RowData data, String key) {
        // 更新操作
        String id = null;
        if (data.getBeforeColumnsList().isEmpty()) {
            return;
        }
        for (CanalEntry.Column column : data.getBeforeColumnsList()) {
            if (column.getIsKey()) {
                id = column.getValue();
                break;
            }
        }
        if(id ==null){
            return;
        }
        String cacheKey =getKey(key).concat(id);
        boolean b = RedisUtil.hasKey(cacheKey);
        if(!b){
            return;
        }
        List<CanalEntry.Column> afterColumnsList = data.getAfterColumnsList();
        JSONObject jsonObject = new JSONObject();
        for (CanalEntry.Column column : afterColumnsList) {
            jsonObject.put(column.getName(), column.getValue());
        }
        RedisUtil.set(cacheKey, jsonObject);
    }

    private static void redisDelete(List<CanalEntry.Column> columns,String name) {
        if(columns.isEmpty()){
            return;
        }
        for (CanalEntry.Column column : columns) {
            if (column.getIsKey()) {
                String cacheKey = getKey(name).concat(column.getValue());
                RedisUtil.del(cacheKey);
            }
        }
    }
    public static String getKey(String name) {
        return "mysql_cache:".concat(name) + ":";
    }
}

bitmap

二进制数组，最大支持2的32次方位

  • 日活统计
  • 连续签到打卡
  • 最近一周活跃用户
  • 统计指定用户一年之中的登录天数
  • 某用户按照一年，哪几天登陆过，哪几天没登录，全年登录的天数

签到

public class SignServiceImpl implements SignService {

    public static final String SIGN = "sign:";
    public static final String USER = "user:";
    public static final String DATE = "date:";
    public static final int DAYS = 31;

    @Resource
    private RedisTemplate redisTemplate;
	
    @Override
    public Map<String, Boolean> signInfo(Integer userId, String date) {
        String key = SIGN + USER + userId + ":" + DATE + date;
        BitFieldSubCommands bitFieldSubCommands = BitFieldSubCommands.create().get(BitFieldSubCommands.BitFieldType.unsigned(DAYS)).valueAt(0);
        List<Long> list = redisTemplate.opsForValue().bitField(key, bitFieldSubCommands);
        Map<String, Boolean> signInfo = new TreeMap<>();
        if (list == null || list.isEmpty()) return signInfo;
        long value = list.get(0) == null ? 0 : list.get(0);
        for (int i = DAYS; i > 0; i--) {
            boolean flag = value >> 1 << 1 != value;
            signInfo.put(date + i, flag);
            value >>= 1;
        }
        return signInfo;
    }

    @Override
    public void signIn(Integer userId) {
        Calendar calendar = Calendar.getInstance();
        int year = calendar.get(Calendar.YEAR);
        int month = calendar.get(Calendar.MONTH) + 1;
        int day = calendar.get(Calendar.DAY_OF_MONTH);
        String key = SIGN + USER + userId + ":" + DATE + year + month;
        redisTemplate.opsForValue().setBit(key, day - 1, true);
        log.info("{}日签到成功", day);
    }

    @Override
    public Map<String, Long> signCount(Integer userId, String date) {
        String key = SIGN + USER + userId + ":" + DATE + date;
        log.info("key = {}", key);
        Long count = (Long) redisTemplate.execute((RedisCallback<Long>) connection -> connection.bitCount(key.getBytes()));
        HashMap<String, Long> map = new HashMap<>();
        map.put(key, count);
        return map;
    }

}

布隆过滤器

一种类似set的数据结构，很长的二进制数组。不保存数据，只是记录标记位

他会将存入的值进行多次hash计算数组下标（可以设置hash次数），然后将此下标位置上的值改为1。通过判断下标上是否有0来判断是否存在

如果有0肯定不存在，如果没0，不一定存在（误差）。

  • 布隆过滤器会出现误差，当下标都是1时，也不一定是存在的，原因是可能存入的数据共享了这个数组下标上的值。所以它并不是完全精确的

    怎么尽量避免误差的出现呢？

    答：增加数组长度或设置多次hash，因为每次hash就相当于一个认证，多次认证后可以提升准确度

    如果存入的值删除了怎么更新布隆过滤器

    答：没有办法直接更新，原因是数组中的值可能是多个存入的值共享的。比较常见的解决方案是，开启一个定时任务，去定期重新创建布隆过滤器

    布隆过滤器误差率能自己设置，那能不能设置成0.000001、

    布隆过滤器误差率和比特位数组长度密切相关，长度越长，发生hash碰撞概率自然就低。所以误差率过低，会导致性能低及资源浪费。

8.1过期策略

Redis的删除不是真的立即删除，

  • 每100ms（默认）抽一部分key来检查是否被删除，发现被标记删除才会删除
  • 查询key，发现key被标记删除，才会删除

这种模式虽然节约了CPU，但是难免出现漏网之鱼，如果大量过期key没有删除，极大浪费内存。所以Redis引入了内存淘汰机制

8.2内存淘汰机制

  1. noeviction：直接报错，不淘汰
  2. allkeys-lru：对key使用lru算法，删除最近最少使用的key，用以保存新数据
  3. volatile-lru：对带过期时间的key使用lru算法，删除带过期时间并且最近最少使用key的
  4. allkeys0random：对所有key随机删除
  5. volatile-random：对所有设置了过期时间的key随机删除
  6. volitale-ttl：删除马上要过期的key
  7. allkeys-lfu：对所有key使用lfu算法进行删除
  8. volatile-lfu：对所有设置了过期时间的key使用lfu算法进行删除

LRU和LFU的区别？

  • LRU：最近最少使用页面置换算法，淘汰最长时间未被使用的页面，看页面最后一次被使用到发生调度的时间长短，首先淘汰最长时间未被使用的页面
  • LFU：最近最不常用的页面置换算法，淘汰一定时间内被访问次数最少的页面，看一定时间段页面被使用的频次，淘汰一定时间内被访问次数最少的页面

LRU主要看时间，LFU主要看频次
